Руководство пользователя

Как работать с системой учёта ИТ-оборудования

Вход в систему

Откройте браузер и перейдите по адресу, который выдал администратор (например https://192.168.1.50:3443). Браузер может показать предупреждение о сертификате — нажмите «Дополнительно» → «Перейти на сайт», это нормально.

⚠ Смените PIN после первого входа. Дефолтный PIN: admn0000. Система напомнит об этом сама при входе.

Реестры оборудования

Всё оборудование разделено на три вкладки:

ВкладкаЧто включает
📋 ОСКомпьютеры, ноутбуки, серверы, моноблоки
📦 МелочиПериферия: мышки, клавиатуры, гарнитуры, веб-камеры
🌐 ИнфраСетевое оборудование: свитчи, роутеры, точки доступа

Добавление оборудования

Нажмите кнопку «+ Добавить» на нужной вкладке. Заполните поля:

ПолеОписаниеОбязательно
ТипТип устройства (ноутбук, ПК, принтер...)✅
МодельНазвание модели✅
Серийный №Уникальный серийный номер с наклейки устройства✅
ОрганизацияЮридическое лицо-владелец✅
ФилиалОфис или подразделение✅

СтатусВ работе / Хранение / Списано / Ремонт✅

Перемещение оборудования

Откройте карточку устройства → кнопка «Переместить». Выберите новый филиал, расположение и ответственного. Все перемещения автоматически фиксируются в истории.

Списание оборудования

Откройте карточку устройства → измените статус на «Списано». Устройство остаётся в реестре, но помечается как выведенное из эксплуатации. История сохраняется.

Поиск и фильтрация

Используйте строку поиска вверху — ищет по модели, серийному номеру, инв. номеру, ответственному. Фильтры позволяют сузить список по организации, филиалу, статусу, типу устройства.

История изменений

Раздел «История» показывает все действия: добавление, перемещения, смену статуса, изменение полей. Можно фильтровать по дате, сотруднику, типу действия.

QR-коды

В карточке устройства есть кнопка «QR-код» — генерирует QR с инвентарным номером для наклейки на устройство. Распечатайте и приклейте для быстрой идентификации.

Экспорт данных

Кнопка «Экспорт CSV» на вкладке реестра — выгружает все видимые устройства в файл CSV, совместимый с Excel.

Импорт из CSV

⚙️ Настройки → Импорт из CSV. Файл должен содержать заголовки:

Вкладка, Инв. номер, Филиал, Расположение, Ответственный, Тип, Модель, Серийный №, Статус, Организация, Примечание, MAC, IP, Hostname, Прошивка

Дубли по серийному номеру пропускаются автоматически.

Управление пользователями

⚙️ Настройки → Пользователи — только для администратора. Можно создавать, редактировать и удалять пользователей, назначать роли, сбрасывать PIN.

Смена PIN

Каждый пользователь может сменить свой PIN в настройках профиля. Минимальная длина — 4 символа. PIN хранится в зашифрованном виде (bcrypt).

Дефолтные учётные данные: логин admin, PIN admn0000. Обязательно смените после первого входа.

Установка на Windows

1. Установите Node.js LTS

2. Скачайте и распакуйте архив с приложением

3. Дважды кликните INSTALL.bat — установит зависимости

4. Дважды кликните START.bat — запустит сервер и откроет браузер

Для автозапуска при старте Windows создайте ярлык START.bat в папке shell:startup

Установка на Linux / macOS

sudo apt update && sudo apt install nodejs npm   # Ubuntu/Debian
chmod +x install.sh start.sh
./install.sh     # установить зависимости
./start.sh       # запустить сервер

Установка Node.js через nvm (рекомендуется)

curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
source ~/.bashrc
nvm install --lts
nvm use --lts

Доступ из локальной сети

Сервер автоматически слушает на всех интерфейсах (0.0.0.0). Узнайте IP сервера:

hostname -I | awk '{print $1}'    # Linux
ipconfig                           # Windows

Сообщите сотрудникам адрес: https://192.168.x.x:3443

Открыть порт в firewall

sudo ufw allow 3443/tcp && sudo ufw reload         # Ubuntu (ufw)
sudo firewall-cmd --add-port=3443/tcp --permanent   # RHEL/Fedora
sudo firewall-cmd --reload

Автозапуск как systemd-сервис (Linux)

[Unit]
Description=IT Assets
After=network.target

[Service]
WorkingDirectory=/opt/it-assets
ExecStart=/usr/bin/node server/index.js
Restart=always
User=www-data
Environment=NODE_ENV=production

[Install]
WantedBy=multi-user.target
sudo cp it-assets.service /etc/systemd/system/
sudo systemctl enable --now it-assets

Данные хранятся в папке data/. Для бэкапа достаточно скопировать эту папку:

cp -r /opt/it-assets/data /backup/it-assets-$(date +%Y%m%d)

Обновление системы

1. Остановите сервер

2. Сделайте бэкап папки data/

3. Замените файлы server/ и public/ новыми

4. Запустите npm install

5. Запустите сервер — миграция схемы произойдёт автоматически

Данные не затрагиваются при обновлении — папка data/ не меняется, схема мигрирует автоматически при первом запуске новой версии.

Что делает система

Веб-приложение для учёта ИТ-оборудования внутри организации. Ведёт реестр устройств (ноутбуки, ПК, принтеры, сетевое оборудование и др.), отслеживает перемещения, назначает ответственных, ведёт справочник сотрудников, генерирует инвентарные номера, экспортирует/импортирует данные в CSV, делает резервные копии и генерирует QR-коды для инвентарных номеров.

Docs / Архитектура

Архитектура системы

SPA-приложение на Node.js с SQLite (встроенный node:sqlite). Backend разбит на repo/routes слои после рефакторинга фаз 1–4d, база данных переведена с lowdb на SQLite в Фазе 7c.

Схема компонентов

🌐 Браузер — SPA
JS разбирается на файлы (Фаза 5, в процессе: 4806 → 3428 строк, 13 файлов вынесено в public/js/). Это classic-скрипты (<script src>), НЕ type="module" — в конце index.html есть синхронный вызов render() до DOMContentLoaded, а ES-модули выполняются deferred. Стили — в public/css/main.css. Обращается к бэкенду через fetch/XHR.
↕ HTTPS (redirect с HTTP)
🖥 Express Server — server/index.js
Монтирует Router-ы из server/routes/ + раздача статики из public/. HTTP на 3000 (редирект), HTTPS на 3443. После рефакторинга: ~548 строк вместо 1785.
↕ require()
🗄 Database Layer — server/database.js + server/db/sqlite.js
С Фазы 7c: вся прикладная данные (активы, история, организации, филиалы, локации, сотрудники, пользователи, аккаунты, настройки, справочники) — в SQLite через встроенный node:sqlite (без нативных аддонов, ноль пересборки под платформу). lowdb (db/store.js) остался только для служебной _meta.schema_version — используется исключительно как источник для одноразовой автомиграции в SQLite при первом запуске.
↕ prepared statements (node:sqlite)
📁 data/
it-assets.sqlite — вся прикладная база (assets, history, organizations, org_inv_rules, filials, locations, employees, users, accounts, settings, categories, type_codes)    |    db.json / config.json — легаси lowdb-файлы, только служебные метаданные схемы    |    cert.pem / key.pem — TLS    |    backups/ — резервные копии (ZIP, включают все три файла)    |    logs/ — структурированные JSON-логи с ротацией по дате

Ключевые принципы

SQLite, WAL-режим. С Фазы 7c данные пишутся через node:sqlite (встроен в Node.js, без нативных аддонов). WAL (Write-Ahead Logging) даёт конкурентное чтение во время записи — важно при нескольких пользователях в LAN одновременно. Каждая мутация — отдельный `INSERT`/`UPDATE` через prepared statement; составные операции (например, ликвидация организации с переносом активов) обёрнуты в явную транзакцию (`BEGIN`/`COMMIT`/`ROLLBACK`), так как `node:sqlite` не имеет встроенного помощника наподобие `better-sqlite3`'s `.transaction()`.
Легаси lowdb-файлы. db.json/config.json остаются на диске, но с Фазы 7c несут только служебную _meta.schema_version — использовались как источник для одноразовой автомиграции в SQLite при первом запуске после обновления. Прикладной код их больше не читает и не пишет.
Мягкое удаление. Assets не удаляются — им ставится status: 'списан'. Все выборки фильтруют a.status !== 'списан'. Сотрудники аналогично не удаляются, а деактивируются (active:false).
Системные заглушки. В справочниках есть записи sys-org-unk, sys-filial-unk, sys-location-unk с флагом system: true. API их скрывает, они нужны для ассетов без привязки к орг/филиалу. Системный пользователь sys-user-admin нельзя удалить или сменить ему роль.
Версионируемая lowdb-схема (легаси). server/migrate.js запускается автоматически при каждом старте (внутри server/db/store.js) и последовательно применяет идемпотентные шаги миграции lowdb-данных до текущей версии — это происходит ДО того, как SQLite-слой (server/db/sqlite.js) читает эти данные для автомиграции, поэтому SQLite всегда получает уже нормализованные данные. После первого запуска на актуальной схеме — no-op при каждом следующем старте. Ручной перезапуск — POST /api/migrate.
HTTPS обязателен. С этой версии сервер поднимает HTTPS на 3443 с самоподписанным сертификатом (server/cert.js), а HTTP на 3000 отдаёт только 301-редирект на HTTPS. Если сертификат сгенерировать не удалось — фолбэк на обычный HTTP на 3000 (см. консольный вывод при старте).

Авторизация (три режима)

Middleware requireAuth поддерживает варианты входа:

РежимЗаголовки / BodyОписание
Пользователь системы (PIN)x-user-id: UUID + x-edit-password: PINВход через Users API. Роль viewer не может мутировать данные (403).
Логин + парольPOST /api/users/login — {login, password}Альтернативный вход по полю login пользователя (сверяется с pin как паролем). Возвращает того же пользователя, что и вход по PIN.
Единый пароль (legacy)x-edit-password: парольРаботает только в паре с существующим x-user-id. Отдельный эндпоинт /api/auth для проверки «сырого» пароля без пользователя удалён.

Эндпоинты с 👑 admin дополнительно проверяются requireAdmin — нужна роль admin.

Rate limiting на вход. /api/users/auth и /api/users/login ограничены 10 неудачными попытками за 5 минут с одного IP; при превышении — блокировка на 15 минут (429 + заголовок Retry-After). Успешный вход сбрасывает счётчик. IP берётся из TCP-соединения (req.socket.remoteAddress), а не из заголовка X-Forwarded-For — иначе клиент мог бы подделать заголовок и обойти лимит. Для деплоя за реверс-прокси см. TRUST_PROXY в разделе «Запуск и конфигурация».
Security-заголовки и обработка ошибок. X-Powered-By отключён (app.disable('x-powered-by')). На каждый ответ проставляются X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Referrer-Policy: no-referrer. CSP сознательно не выставлен — фронтенд активно использует inline-обработчики событий (onclick и т.п.), строгий CSP сломает его без рефакторинга фронтенда. В конце цепочки middleware стоит общий error-handler: любое необработанное исключение (например, синтаксически неверный JSON в теле запроса) логируется на сервере полностью, а клиенту отдаётся только краткое сообщение без stack trace и путей на диске.

Авторизация

Режим 1 — пользователи системы (PIN): x-user-id: UUID + x-edit-password: PIN
Режим 2 — логин + пароль: POST /api/users/login с {login, password}
Режим 3 — единый пароль (legacy): x-edit-password: пароль вместе с валидным x-user-id

Users API

Пользователи системы с ролями и PIN. Хранятся в it-assets.sqlite → users (SQLite, с Фазы 7c).

Роли: admin — полный доступ, управление пользователями · operator — редактирование данных · viewer — только чтение (все мутирующие запросы → 403)
sys-user-admin — системный пользователь (login: admin, pin: admn0000 по умолчанию). Нельзя удалить и нельзя сменить роль/деактивировать через API; менять name/login/pin/email — можно.
Rate limit на вход: /api/users/auth и /api/users/login — 10 неудачных попыток / 5 мин с одного IP, затем блокировка 15 мин (429).

Employees API

Справочник сотрудников — источник значений поля responsible у Asset. Хранится в it-assets.sqlite → employees (SQLite, с Фазы 7c).

Это не то же самое, что Users (пользователи системы с ролями). Employee — просто карточка сотрудника (ФИО, отдел, филиал, телефон), на которую ссылается оборудование.

Список / поиск сотрудников

Без явного requireAuth, но проверяет, что x-user-id указывает на активного пользователя (401 иначе). Если передан q (мин. 2 символа) — возвращает результаты поиска по name/dept/phone (макс. 15). Иначе — полный список; active=false включает деактивированных.

/api/employees/:id
Обновить 🔐 auth

Разрешённые поля: name, dept, filial, phone, email, note, active.

POST
/api/import/history
Импорт истории из CSV/Excel 🔐 auth

Body: { "rows": [...] }. Поддерживает Excel-сериальные даты (44000+).

Авто-сопоставление с активами по серийному номеру. Поле equipment в источнике часто описывает несколько предметов через запятую (реальный пример из исходных Excel-выгрузок: "Ноутбук HUAWEI MCLF-X SN 28UBB..., Монитор XIAOMI ... SN 52756/..."). Каждая позиция разбивается на отдельную запись истории. Серийник извлекается трёхуровневым фоллбеком (метка SN/sn/SN: → таб-разделённый последний сегмент → последнее «слово» в тексте, каждый кандидат проверяется на «похожесть на серийник»: без пробелов, есть цифра, 6–25 символов) и сверяется на точное совпадение с реальными активами в базе. Совпадение — asset_id/model/type/filial/location подтягиваются из актива, а не из текста. Без совпадения (или без серийника вовсе, например для аксессуаров) — запись всё равно сохраняется, просто без привязки.

Дедупликация — на уровне отдельной позиции (после разбивки), по ключу date|позиция|from_who, а не по исходной строке целиком.

Backup API

Резервное копирование it-assets.sqlite + db.json + config.json в data/backups/.

Автоматически: раз в час (setInterval, тег auto) и один раз через 10 секунд после старта сервера (тег startup). Плюс автоматический бэкап перед восстановлением из другого файла (тег pre-restore). Отключено при NODE_ENV=test (Jest выставляет это значение автоматически) — иначе таймеры реально писали бы zip-файлы на диск при каждом прогоне тестов.
Ротация: хранится не более 30 последних файлов (pruneBackups(30)) — старые удаляются автоматически при каждом создании нового.
Формат: если установлен пакет adm-zip — единый ZIP-архив с it-assets.sqlite, db.json и config.json внутри (backup_<tag>_<timestamp-с-мс>_<rnd6>.zip). Перед архивацией SQLite делает PRAGMA wal_checkpoint(TRUNCATE) — иначе часть данных могла бы остаться только в -wal-файле, который в бэкап не попадает. Без adm-zip — fallback на пару JSON-файлов (db.json + одноимённый .config.json; в этом режиме it-assets.sqlite не бэкапится — известное ограничение fallback-режима). Штамп времени включает миллисекунды, плюс добавлен случайный 6-символьный суффикс — раньше штамп обрезался до секунд, и два бэкапа в одну секунду молча перезаписывали друг друга.
Расположение: <IT_ASSETS_DATA_DIR или data/>/backups/ — путь читается напрямую из файловой системы (fs.*), в обход database.js. См. переменную IT_ASSETS_DATA_DIR в разделе «Запуск и конфигурация» — важно для тестов и для нестандартного расположения данных.

Сервер поднимает два листенера:

ПортПротоколНазначение

Сертификат генерируется один раз при первом запуске и переиспользуется дальше (server/cert.js → ensureCert()). Фронтенд

JS-логика разбирается на файлы (Фаза 5, в процессе, см. дорожную карту): 13 файлов уже вынесены в public/js/ (ui-utils.js, qr.js, theme.js, i18n.js, auth.js, meta-fields.js, global-search.js, router.js, views/{dashboard,alerts,accounts,asset-tab}.js) — это classic-скрипты (<script src>), не ES-модули: в конце index.html есть синхронный render(); до DOMContentLoaded, а модули выполняются deferred, поэтому функции вроде esc()/toast() были бы undefined в момент первого рендера. Файлы делят один глобальный scope с оставшимся кодом — имена резолвятся в момент вызова, не объявления. Ещё в index.html: renderHistory, renderSettings (крупнейший экран — orgs/filials/locations/users/type-codes/backup), формы создания/редактирования актива. Стили — отдельно, public/css/main.css. Express раздаёт public/ как статику. Fallback-роут GET * тоже отдаёт index.html (SPA-режим).

E2E тесты (Playwright)

Jest-тесты (283 шт.) бьют только по API — ни один не проверяет, что реально рисуется в браузере. Playwright закрывает этот пробел: открывает настоящий Chromium, кликает по интерфейсу, проверяет видимое поведение. Это подстраховка специально для Фазы 5 (разбор public/index.html на файлы) — правило то же, что и с Jest: гоняем после каждого вынесенного JS-модуля, число прошедших сценариев не должно меняться.

npx playwright install chromium   # один раз, скачивает браузер

npm run test:e2e:ui               # с UI-раннером Playwright
./test-e2e.sh                     # Linux/macOS

Изоляция

playwright.config.js сам поднимает сервер (node server/index.js) на отдельном порту (3543) с изолированной data-директорией .e2e-data/ через IT_ASSETS_DATA_DIR — та же переменная окружения, которую уважают server/db/store.js и server/cert.js. Реальные данные и сертификат проекта не трогаются. Каждый прогон стартует с чистого состояния (админ admin / admn0000 — дефолтный PIN).

Найденные при подготовке E2E грабли (для будущих сценариев)

Неоднозначные селекторы по тексту. button:has-text("Войти") совпадает сразу с несколькими кнопками на странице (#auth-btn, кнопка в модалке и т.д.) — Playwright молча берёт первую попавшуюся. Использовать точные id или onclick-атрибут (например, button[onclick="doLogin()"]), не текст.
Nag-модалка "Смените пароль". При входе под настоящим дефолтным PIN (admn0000) через 600мс сама система показывает модалку смены пароля (_showDefaultPinWarning), которая перехватывает все последующие клики. Хелпер login() в тесте её закрывает («Напомнить позже»), если она появилась.
Логаут — мгновенный, без модалки. toggleAuth() при активной сессии делает выход сразу же (if (canEdit()) { ...; render(); return; }), а не открывает форму логина заново — проверять нужно скрытие вкладок, не появление #m-login.
Известный (не наш) баг, найденный при переносе кода в Фазе 5: битый onclick у кнопки "→" в результатах глобального поиска (public/js/global-search.js) — похоже, event.stopPropagation() был случайно разорван вставкой showDetail(...). Перенесён как есть при рефакторинге (без изменения поведения), почему чинить нужно отдельным PR, не смешивая с рефакторингом.
